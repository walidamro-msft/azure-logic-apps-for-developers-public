Detailed Training Guide

A condensed talk track based on the attached deck guidance. Use it as a facilitation script, not as attendee-facing reading material.

Trainer-onlyFacilitation guide

Overview Talk Track

  • Explain Logic Apps as workflow automation across systems, not simply as a connector gallery.
  • Stress the value of eliminating manual handoffs and reducing bespoke integration code.
  • Use the workflow-structure slide to anchor trigger first, then action chain, then control flow.

Core Concepts Talk Track

  • Triggers define when a workflow starts and therefore drive both responsiveness and cost.
  • Actions are the units of business work; connectors are the units of external reach.
  • When covering stateful versus stateless, tie the choice to durability, performance, and run-history needs.
  • Distinguish Local Functions from external Function Apps: local .NET functions deploy with a Standard logic app and need no separate service plan, while external Function Apps use Azure Functions pricing.
  • Power Automate is subject to license-based Power Platform request limits. Logic Apps does not use those allocations, although Consumption throughput limits and connector or downstream throttling still apply.

Advanced Patterns Talk Track

  • Conditions and loops make workflows adaptive rather than linear.
  • Retry and failure routing are first-class design responsibilities, not afterthoughts.
  • Azure Functions and Service Bus should be presented as extension points, not competing products.
  • Built-in connectors run within the Logic Apps compute environment and generally have lower latency than managed connectors. Connector Namespace exposes managed connectors only.

Pricing and Plans Talk Track

  • Start with the deck's mental model: Consumption is pay-per-execution, Standard is fixed compute.
  • Update the room with the current Learn framing: Standard also means single-tenant hosting, multiple workflows, stronger networking, and local engineering workflows.
  • Be explicit that exact limits and pricing should always be revalidated before architecture decisions.

Operations Talk Track

  • Teach run history first because it is concrete and immediate.
  • Then connect that local debugging view to Azure Monitor and Application Insights for platform-scale observability.
  • Reinforce that idempotency matters because integration platforms are naturally at-least-once.

Security and Governance Talk Track

  • Frame RBAC around separation of design, operations, and audit roles.
  • Use limits to explain why workflow architecture is not only about business logic.
  • Position Standard as the likely answer when networking and private access dominate requirements.

Tooling Talk Track

  • Use the portal screenshot to onboard the room visually.
  • Use the VS Code screenshot to show that Logic Apps also supports source-controlled engineering workflows.
  • Call out the separate Consumption and Standard extensions so attendees do not assume one tool path fits both models.
