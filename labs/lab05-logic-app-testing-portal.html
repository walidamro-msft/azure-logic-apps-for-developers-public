Lab 05: Logic App Testing

Mock an HTTP action to validate success and failure behavior without calling a live product service.

Azure portalConsumption or Standard~35 min

Scenario and Objectives

A workflow depends on a product service that is unavailable during development. You will configure static results on the HTTP action, verify two control-flow paths, and then restore live behavior.

  • Build a small HTTP-request workflow.
  • Mock a successful downstream response.
  • Mock a failed action and verify failure handling.
  • Disable static results before completing the lab.
Limit: static results apply to actions, not triggers. Trigger the workflow normally and mock only the downstream HTTP action.

Build the Test Workflow

  1. Create or open a lab Logic App workflow in the portal.
  2. Add When an HTTP request is received, method POST, with a body containing a string property named sku.
  3. Add an HTTP action named Product lookup. Set method GET and URI to https://example.invalid/products/@{triggerBody()?['sku']}. The reserved invalid domain ensures a live call cannot reach a real service.
  4. Add a Response action named Product found, status 200, with the body below.
  5. Add a Response action named Dependency unavailable, status 503. Configure its Run after to run only when Product lookup has failed or timed out.
  6. Configure Product found to run only when Product lookup succeeds. Save the workflow.
{
  "requestedSku": "@{triggerBody()?['sku']}",
  "product": "@{body('Product_lookup')}",
  "testRun": "@{workflow()?['run']?['name']}"
}

Test a Successful Action

  1. Select Product lookup and open its Testing tab.
  2. Select Enable Static Result.
  3. Set action status to Succeeded. Configure outputs with status code 200, content-type application/json, and the body below.
  4. Save, trigger the workflow with { "sku": "AZ-100" }, and inspect the run.
{
  "sku": "AZ-100",
  "name": "Workflow Mug",
  "price": 18.50,
  "inventory": 24
}
  • The action displays the static-result beaker indicator.
  • The workflow returns HTTP 200 with the mocked product.
  • Run details show Product lookup used a static result.

Test a Failed Action

  1. Return to the Product lookup Testing tab and edit its static result.
  2. Set status to Failed and provide an error code such as ProductServiceUnavailable with a clear test message.
  3. Save and submit the same trigger payload.
  4. Confirm Product found is skipped and Dependency unavailable returns HTTP 503.

What to inspect

Open each action in run details. Verify the mocked error is visible, the run-after rule selected the failure path, and no outbound request reached the invalid URI.

Restore Live Behavior

  1. Select Product lookup, open Testing, and choose Disable Static Result.
  2. Save the workflow and confirm the beaker indicator is gone.
  3. Do not run against a production endpoint until authentication, authorization, retry, timeout, and test-data controls are ready.
Important: disabling static results can preserve the configured mock values for reuse. The absence of the enabled indicator is what confirms the action will call its configured endpoint.

Troubleshooting

SymptomCheck
Testing tab is unavailableStatic results are action-specific; select the HTTP action rather than the trigger or a control.
Success response is skippedCheck the mocked action status and the Response run-after settings.
Workflow calls the URIConfirm static results are enabled and the action shows its test indicator before running.

Cleanup and References

Disable static results, disable or delete the test workflow, and remove the lab resource group when finished.
