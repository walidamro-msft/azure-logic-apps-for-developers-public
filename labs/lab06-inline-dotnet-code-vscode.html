Lab 06: Inline .NET Code

Use a workflow-local C# script to validate line items and calculate subtotal, tax, and total.

Visual Studio CodeStandard only~50 min

Scenario and Objectives

An order workflow receives a variable number of line items. The calculation is small and belongs only to this workflow, but nested loops and expressions would obscure the business rule. You will implement the rule in the Execute C# Script Code action.

  • Read trigger data through WorkflowContext.
  • Validate items and calculate currency values with decimal.
  • Return a typed result to the next workflow action.
  • Recognize when custom logic should move to Azure Functions.

Create the Workflow

  1. Create a Logic Apps Standard project with a Stateful Workflow named inline-pricing.
  2. Add When a HTTP request is received, set method to POST, and paste the schema below.
  3. Add the Inline Code Operations action Execute C# Script Code. Rename its display name to Calculate pricing. Logic Apps generates the internal action name Calculate_pricing by replacing the space with an underscore. The designer shows the display name, while workflow expressions use the internal name.
  4. Replace the sample code file with the C# script in the next section. Keep the generated .csx file beside the workflow's workflow.json.
  5. Add a Response action after the script. Set status to 200 and body to @body('Calculate_pricing'). Here, Calculate_pricing is the internal action name from step 3.
  6. Add a second Response action with status 400 (for error handling). Right-click it and select Run after. In the Settings panel:
    1. Click Select actions and choose Calculate pricing, the display name shown in the designer (remove Response if present).
    2. Uncheck all conditions except has failed.
    3. Switch to the Parameters tab and set the response Body to @outputs('Calculate_pricing'), using the internal action name, to include error details.

Add the C# Pricing Script

Understand the script contract

A Logic Apps C# script is a .csx file compiled and hosted by the Azure Logic Apps runtime. The runtime supplies the workflow objects and invokes the script entry point.

Code elementPurpose
RunThis predefined method name is the script entry point. The runtime calls Run, so do not rename it. Use Task<T> with async when awaiting workflow data and returning a value.
WorkflowContext contextThe runtime provides this object. Use GetTriggerResults() for the trigger or GetActionResults("action-name") for a preceding action. Each operation result can include inputs, outputs, status, timing, tracking information, and error details.
ILogger logThe runtime injects this logger. Use it instead of Console.Write so messages flow through the host logging pipeline and can appear in Application Insights when configured. Log identifiers and counts rather than sensitive payloads.
#r directivesA #r directive tells the script compiler to reference an assembly. Here, the script references the JSON and Logic Apps scripting assemblies available to the host.
using directivesA using directive imports a namespace from a referenced or host-provided assembly, allowing names such as WorkflowContext, ILogger, and JObject without their fully qualified names.
PricingResultsThis class defines the shape returned by Run. The runtime serializes the instance to JSON and places it in the C# action's output body for later workflow actions.
Assembly versus namespace: #r "Microsoft.Azure.Workflows.Scripting" makes the assembly available to the script compiler; using Microsoft.Azure.Workflows.Scripting; then imports its namespace. The Logic Apps host already provides common framework and logging assemblies, while these explicit directives make this script's dependencies clear.
// Reference assemblies used by this .csx script.
#r "Newtonsoft.Json"
#r "Microsoft.Azure.Workflows.Scripting"

// Import namespaces so their types can be used without full names.
using Microsoft.Azure.Workflows.Scripting;
using Microsoft.Extensions.Logging;
using Newtonsoft.Json.Linq;

// Logic Apps calls this predefined Run method and supplies context and log.
public static async Task<PricingResults> Run(WorkflowContext context, ILogger log)
{
  // Read the request trigger output and require a JSON object in its body.
    var triggerOutputs = (await context.GetTriggerResults().ConfigureAwait(false)).Outputs;
    var body = triggerOutputs?["body"] as JObject
        ?? throw new ArgumentException("The request body is required.");

  // Extract the fields used by the pricing rule.
    var orderId = body.Value<string>("orderId");
    var items = body["items"] as JArray;
    var taxRate = body.Value<decimal?>("taxRate") ?? 0.08m;

  // Reject incomplete requests before calculating a result.
    if (string.IsNullOrWhiteSpace(orderId))
        throw new ArgumentException("orderId is required.");
    if (items == null || items.Count == 0)
        throw new ArgumentException("At least one line item is required.");
    if (taxRate < 0m || taxRate > 1m)
        throw new ArgumentException("taxRate must be between 0 and 1.");

    // Visit each object in the items array, validate it, and add its line total.
    decimal subtotal = 0m;
    foreach (var item in items)
    {
        var sku = item.Value<string>("sku");
        var quantity = item.Value<int>("quantity");
        var unitPrice = item.Value<decimal>("unitPrice");

        if (string.IsNullOrWhiteSpace(sku) || quantity <= 0 || unitPrice < 0m)
            throw new ArgumentException("Each item needs a SKU, positive quantity, and nonnegative unitPrice.");

        subtotal += quantity * unitPrice;
    }

      // Round currency values once the line-item loop is complete.
    subtotal = Math.Round(subtotal, 2, MidpointRounding.AwayFromZero);
    var tax = Math.Round(subtotal * taxRate, 2, MidpointRounding.AwayFromZero);

      // Send operational information through the host logger without the payload.
    log.LogInformation("Calculated pricing for order {OrderId} with {ItemCount} items.", orderId, items.Count);

      // The runtime serializes this object into the action output body.
    return new PricingResults
    {
        OrderId = orderId,
        ItemCount = items.Count,
        Subtotal = subtotal,
        Tax = tax,
        Total = subtotal + tax,
        Currency = "USD"
    };
}

// This return model gives downstream actions a predictable JSON shape.
public class PricingResults
{
    public string OrderId { get; set; }
    public int ItemCount { get; set; }
    public decimal Subtotal { get; set; }
    public decimal Tax { get; set; }
    public decimal Total { get; set; }
    public string Currency { get; set; }
}

Follow the data through the script

  1. GetTriggerResults() retrieves the Request trigger result, and Outputs["body"] selects the submitted JSON body.
  2. The script reads orderId, items, and taxRate. Because taxRate is optional, the null-coalescing operator (??) supplies 0.08m when it is absent.
  3. The foreach loop processes one JSON object from the items array at a time. It reads that item's SKU, quantity, and unit price, validates them, then adds quantity * unitPrice to subtotal.
  4. After every item has been processed, the script rounds the subtotal, calculates and rounds tax, and writes a structured log message.
  5. The script creates one PricingResults object. Its public properties become JSON fields in the action output body, which the 200 Response action reads with @body('Calculate_pricing').
Why use a result class? Returning a typed model documents the output contract, keeps all response fields together, and gives downstream actions a stable JSON shape. If you add or remove a public property in PricingResults, you change the data exposed by the script action.

Run and Verify

  1. Start Azurite if your local project requires it, then press F5.
  2. Send the valid request to the local callback URL.
  3. Confirm the response values match the expected result.
  4. Send the invalid request and confirm the script fails and the 400 response path runs.
  5. Inspect local run details and the host log. Confirm the log identifies the order and item count without logging the complete customer payload.

Valid request

{
  "orderId": "ORD-2001",
  "taxRate": 0.08,
  "items": [
    { "sku": "AZ-100", "quantity": 2, "unitPrice": 18.50 },
    { "sku": "AZ-200", "quantity": 1, "unitPrice": 25.00 }
  ]
}
{
  "orderId": "ORD-2001",
  "itemCount": 2,
  "subtotal": 62.00,
  "tax": 4.96,
  "total": 66.96,
  "currency": "USD"
}

Invalid request (to test 400 error response)

{
  "orderId": "ORD-2002",
  "taxRate": 0.08,
  "items": [
    { "sku": "AZ-100", "quantity": 0, "unitPrice": 18.50 }
  ]
}

This request has a quantity of 0, which violates the validation rule. The pricing script will throw an exception, Response 1 will execute (because Calculate_pricing has failed), and you'll see the error message in the 400 response body.

Inline Code or Azure Functions?

Keep inline C# whenMove to Azure Functions when
The logic is small, workflow-local, and executes within action limits.Multiple workflows or applications share the logic.
Deployment and versioning should follow one workflow.The code needs independent deployment, scaling, testing, or ownership.
The script is clearer than a large expression graph.The solution needs long execution, large output, complex dependencies, or a public API.

Troubleshooting

SymptomCheck

Body is nullInspect trigger outputs and confirm the JSON request used application/json.
