Azure Logic Apps Consumption Plan

A fully managed, multitenant plan that scales automatically and charges for trigger and action executions rather than reserved compute.

Pay per executionOne workflow per resourceMultitenant
Important: All Consumption workflows are stateful. Consumption does not support stateless workflows. If your solution requires stateless execution, choose Logic Apps Standard.

Architecture and Resource Model

Each Consumption logic app resource contains one workflow. The Azure-managed multitenant runtime supplies compute, scale, storage, and regional service endpoints, so there is no instance size to select or idle compute charge.

Execution Billing

Billing follows operation executions, not workflow duration. A trigger poll, action attempt, loop cycle, and retry can each add an execution. Failed and skipped trigger checks can still be metered.

Counting Patterns That Change Cost

Polling

Count every scheduled check, not only checks that start runs. One-minute polling is about 43,800 checks in a 30.4-day month.

Split On

An array returned by a trigger can create one workflow instance per item, multiplying downstream operations. Its limit is 100,000 items with trigger concurrency off, but only 100 items after concurrency is enabled.

For each

Count the loop operation plus each inner action for every item processed.

Retries

Count the initial attempt and every retry. Five retries means six executions.

Conditions

Count the condition and only actions on the branch that actually runs. Actions skipped because another branch was selected are not billed.

Pagination and chunking

Consumption normally meters the operation execution rather than each underlying call, even when pagination makes multiple calls.

Networking and Isolation

Consumption uses shared multitenant endpoints and region-specific inbound/outbound addresses. It does not provide the direct VNet integration and private endpoint model of Standard. Managed connectors may run outside the workflow runtime and have their own service tags and outbound addresses.

Important: choose Standard when private access to VNet resources, predictable outbound networking, or single-tenant compute is a hard requirement.

When Consumption Fits

Strong fit

Low or variable traffic, straightforward SaaS automation, event-driven integration, rapid setup, and no private-network requirement.

Workflow model

Choose Consumption only when stateful workflow behavior meets the requirement. All Consumption workflows are stateful; stateless workflows require Standard.

Cost warning

Frequent polling, large loops, retries, and high managed-connector volume can make per-execution pricing exceed Standard compute.

Architecture warning

Do not select Consumption when the solution requires multiple workflows per app, direct VNet integration, local runtime debugging, or regional-only workflow data.

Enable High-Throughput Mode Deliberately

Consumption defaults to 100,000 action executions per five-minute rolling interval. High-throughput mode raises that limit to 300,000 and is currently a preview capability. In the portal, open Workflow settings and turn on High throughput. For an ARM deployment, use:

"runtimeConfiguration": {
  "operationOptions": "OptimizedForHighThroughput"
}
Capacity check: High-throughput mode increases the workflow execution ceiling, but it does not raise connector-specific throttling limits or downstream capacity. Load test the complete path and distribute work across workflows when one workflow becomes the bottleneck.
