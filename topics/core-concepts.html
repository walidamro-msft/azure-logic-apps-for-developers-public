Core Concepts

These are the building blocks that show up in every workflow: how work starts, how steps execute, how integrations connect, how failures are handled, and how operations teams observe runs.

Core moduleArchitecture fundamentals

Triggers Start Every Workflow

Every workflow starts with exactly one trigger. In the Add a trigger pane, you choose the operation that defines when a run begins, then add actions after that trigger.

Use this sequence to inspect the trigger experience in the designer:

  • Select Add a trigger to open the first required step in a new workflow.
  • Open From Azure and Other to compare browsing connector galleries with searching directly.
  • Search by name with examples such as Outlook, Schedule, or HTTP.
  • Filter by connector type: compare Built-in, Shared, and Custom options.
  • Open Recurrence and inspect interval, frequency, time zone, and start time choices.
  • Open When an HTTP request is received and inspect method selection, request schema, and callback URL generation after save.
  • Save the workflow to finalize trigger configuration and endpoint behavior.

Connector event trigger

Start when something changes in a connected service, such as a new email in Office 365 Outlook, a file in SharePoint, a blob in Azure Storage, or an event from another Azure service.

Recurrence trigger

Run on a schedule, such as every few minutes, hourly, daily, or on a business calendar. Use this for polling, cleanup, reporting, and batch synchronization.

HTTP request trigger

Expose the workflow as an endpoint that another app, webhook, approval flow, or integration process can call when work needs to start immediately. The request URL is generated after the workflow is saved.

Configure Trigger Conditions Deliberately

Trigger conditions are filters that Logic Apps evaluates before a run starts. They avoid unnecessary workflow runs and reduce operational noise, but a polling trigger still checks its endpoint on schedule.

  • Set schedule intervals intentionally rather than defaulting to aggressive polling.
  • Use trigger conditions and filters so only relevant events create workflow runs.
  • In Consumption, polling trigger checks are metered even when the trigger is skipped, so use an appropriate interval or a push trigger when cost matters.
  • Standard stateless workflows support push triggers only. Recurrence and managed connector polling triggers require a stateful workflow.

Example

Imagine a support mailbox where you only want Logic Apps to start when an incoming email subject contains Urgent. You can add a trigger condition so the workflow ignores every other message.

@contains(triggerBody()?['subject'], 'Urgent')

The workflow still watches the mailbox, but it creates a run only when the condition returns true. That is the difference between seeing every event and starting only on the events you care about.

Actions Perform the Work

After the trigger fires, actions execute the workflow's business steps. An action might call an API, transform payloads, insert a record, send a message, or branch based on data.

Data operations

  • Compose, select, filter, parse, and transform payloads.
  • Normalize data before pushing it downstream.

Control actions

  • Conditions, switches, loops, scopes, and parallel branches.
  • Model workflow logic without custom orchestration code.

Integration actions

  • Call Microsoft services, third-party APIs, databases, queues, and enterprise systems.
  • Mix built-in and managed connectors as needed.

Example

A simple workflow might start with an HTTP trigger, use Parse JSON to extract an order number, call an HTTP action to look up order details, then use Send an email or Post message in Teams to notify the next team.

Workflow Logic Controls the Path

Workflow logic is where Logic Apps stops being just a sequence of steps and becomes orchestration. This area includes conditions, switch cases, and loop controls that decide what happens next.

If-then with Condition

  • Use a Condition when you need a true/false decision.
  • Typical example: if an order total is greater than a threshold, route to approval; otherwise, continue automatically.

Case logic with Switch

  • Use a Switch when one field can map to multiple outcomes.
  • Typical example: route requests differently for HR, Finance, and IT.

Repetition with loops

  • Use For each for arrays and Until for polling or waiting on state changes.
  • Typical example: send one message per uploaded file, or keep checking an API until status becomes Completed.

Logic Pattern Examples

Important: For each loops run in parallel by default. If a loop updates shared variables or must process records in order, turn on sequential execution.

Inline Code and Script Actions

Azure Logic Apps now refers to this area as Inline Code or Inline Code Operations, depending on the workflow experience. This capability lets you run small code snippets directly inside a workflow when expressions, data operations, and built-in actions are not enough.

ActionLanguageWhere it appliesTypical use
Execute JavaScript CodeJavaScriptConsumption and StandardLightweight parsing, reshaping, pattern matching, or custom payload manipulation.
Execute CSharp Script CodeC# script / .NETStandard onlyMore advanced transformation, file processing, package reuse, or richer custom logic.
Execute PowerShell CodePowerShellStandard onlyAdministrative automation, Azure operations, scripting with modules, or managed identity driven tasks.

JavaScript snippets

  • Best for short inline transformations inside a workflow step.
  • Common example: extract email addresses, normalize JSON, or compute derived values before a later action.

C# scripts

  • Best for Standard workflows that need richer code structure, .csx files, or NuGet-backed logic.
  • Common example: unzip content, encrypt data, aggregate multi-step outputs, or implement logic that is awkward in expressions.

PowerShell scripts

  • Best for Standard workflows that automate Azure or script-friendly tasks.
  • Common example: connect with managed identity, inspect another resource, run module-based commands, and push results back to the workflow.

Local Functions in a Logic Apps Workspace

Standard logic apps can also include .NET custom functions in the same Visual Studio Code workspace and deployment as the workflows that call them. Use the built-in Call a local function in this logic app action to keep workflow orchestration and reusable custom code together.

Cost boundary: local functions that run inside the Logic Apps workspace are included in the Logic Apps hosting cost and do not create a separate Azure Functions charge. A function hosted in a separate Azure Function App is an external dependency and follows the Azure Functions pricing model for its hosting plan.
Learn more: Create and run .NET custom functions from Standard workflows

Quick Examples

// Execute JavaScript Code
const subject = workflowContext.trigger.outputs.body.subject;
return { isUrgent: subject.includes("Urgent") };
// Execute CSharp Script Code
public static async Task<Results> Run(WorkflowContext context)
{
    var triggerOutputs = (await context.GetTriggerResults().ConfigureAwait(false)).Outputs;
    return new Results { Message = "Hello from CSharp action" };
}

public class Results
{
    public string Message { get; set; }
}
# Execute PowerShell Code
$trigger = Get-TriggerOutput
$result = [PSCustomObject]@{ Message = "Hello from PowerShell!" }
Push-WorkflowOutput -Output $result

When to use this feature

Use inline code when the requirement is still workflow-local and relatively small. If the logic becomes shared, long-running, independently versioned, or needs its own scaling model, move that code into a separate service such as Azure Functions instead of growing the workflow into an application host.

Important distinctions: Execute JavaScript Code works in both Consumption and Standard. Execute CSharp Script Code and Execute PowerShell Code are Standard workflow capabilities. PowerShell returns values with Push-WorkflowOutput, while C# scripts use a predefined Run method.

Connector Examples

Built-in connector example

Use the built-in Service Bus, SQL Server, or HTTP operations in Standard when you want lower latency and tighter packaging with the workflow runtime.

Managed connector example

Use Office 365 Outlook, SharePoint, Teams, Salesforce, or Dynamics 365 when you need SaaS integration with provider-managed connectivity.

Custom connector example

Use a custom connector when your organization has an internal API or a vendor API that is not represented well by a built-in or managed connector.

How to think about the choice

Connectors answer two practical questions: What system am I talking to? and Where does that integration run? Built-in connectors are usually the first choice for high-throughput Standard workloads; managed connectors are often the fastest route to SaaS integration breadth.

Scopes Group Related Steps

A Scope is a logical container for related actions. After all actions inside the scope finish, the scope itself gets a status such as Succeeded, Failed, Skipped, or TimedOut. That makes scopes useful for grouping primary work, cleanup, and exception handling.

  • Use a scope to treat several actions as one unit of work.
  • Check the scope status instead of placing separate failure handling after every action.
  • Separate business logic from cleanup or compensation logic.

Scope Flow

flowchart TD
    A[Trigger starts workflow] --> B[Scope: Process order]
    B --> B1[Validate order]
    B1 --> B2[Fulfill order]
    B2 --> B3[Complete order]
    B3 --> C{What is the scope status?}
    C -->|Succeeded| D[Continue success path]
    C -->|Failed or TimedOut| E[Run failure handling]

    class A root;
    class B,B1,B2,B3 app;
    class C decision;
    class D success;
    class E cold;

Example

Create a scope named Process order with actions such as Validate order, Fulfill order, and Complete order. Then add a condition after the scope that checks whether actions('Process_order')?['status'] equals Failed. If true, run the failure branch. If false, continue with the success path.

Why scopes matter: they are the cleanest way to build a try/catch style design in Logic Apps. The first scope performs work; a later action or scope runs when the first scope fails, times out, or is skipped.

Integration Accounts for B2B and Artifact Reuse

An integration account is the Azure resource that stores and manages B2B artifacts used by Azure Logic Apps, especially for enterprise integration scenarios. It acts as the container for partners, agreements, maps, schemas, and certificates.

What it stores

ArtifactWhat it isExample use
PartnersBusiness parties that exchange messagesDefine Contoso and Fabrikam as trading partners.
AgreementsRules for partner communicationModel X12 or AS2 exchange rules between two organizations.
SchemasExpected message structureValidate inbound purchase orders against an XSD.
MapsTransformation definitionsConvert an EDI purchase order into your internal XML or JSON format.
CertificatesSecurity and partner trust materialSupport signed or encrypted B2B message exchange.

When to use an integration account

  • When you implement B2B protocols such as AS2, X12, or EDIFACT.
  • When multiple workflows need to reuse the same maps, schemas, or certificates.
  • When you need governed partner management instead of embedding transformation assets separately in every workflow.

Practical example

An order-processing workflow receives X12 messages from a supplier. The logic app uses an integration account to store the X12 agreement, the XML schema for validation, and the map that converts the partner message into the company's canonical order format before routing the data to ERP.

Important distinction: in Standard workflows, some XML and Liquid artifacts can live directly in the logic app project, so you do not always need an integration account. Use one when you need enterprise B2B features, shared artifact governance, or protocol support across workflows.

Error Handling Is a Design Activity

Logic Apps gives you multiple layers of failure handling. The core mechanisms are retry policies, run after conditions, and scope-based exception handling.

Retry policies

  • Use retries for transient HTTP, connector, or network failures.
  • Common policies are Default, None, Fixed interval, and Exponential interval.

Run after

  • Change successor actions so they run after Failed, TimedOut, or Skipped instead of only after Succeeded.
  • This is the simplest branch-level error path.

Scopes

  • Group primary actions inside a scope, then add follow-up handling based on the scope result.
  • This is the usual pattern for centralized exception handling.

Example handling flow

  1. Call a downstream order API with an HTTP action.
  2. Allow retries for 429 and 5xx style transient failures.
  3. If the call still fails, run a follow-up action using run after = Failed.
  4. Log the failure, capture the correlation ID, and notify the operations team.
"runAfter": {
  "Create_Order": ["Failed", "TimedOut"]
}
Important: use retries only for idempotent or safe-to-repeat operations. If an action creates money movement, inventory changes, or external side effects, careless retries can create duplicates.

Monitoring Answers “Is the Workflow Healthy?”

Monitoring in Logic Apps spans three layers: portal run inspection, Azure Monitor metrics and alerts, and diagnostics logs routed to Log Analytics, Storage, or Event Hubs.

Portal view

Use trigger history and run history first when debugging one workflow execution.

Metrics and alerts

Use Azure Monitor to detect failed triggers, rising latency, throttling, and unusual failure rates.

Diagnostics logs

Use Log Analytics for queries across many runs, workflows, and correlated Azure services.

What to monitor

  • Trigger failures and skipped triggers.
  • Workflow run success and failure rates.
  • Action latency, retries, and throttling behavior.
  • Connection health and downstream service availability.

Concrete example

Set an alert when more than one trigger fails in a five-minute window, then use Log Analytics to query whether the failures came from the same connector, the same workflow version, or the same downstream API.

For a deeper operations walkthrough, see the dedicated Monitoring & Troubleshooting page.

Run History Is Your First Debugging Surface

Run history shows what actually happened for each workflow instance: status, action sequence, inputs, outputs, retries, and any errors. Trigger history complements it by showing every trigger attempt, even when a run never started.

ViewWhat it tells youWhen to use it
Trigger historyWhether the trigger fired, skipped, or failed, plus trigger inputs and outputsUse when the workflow did not start as expected.
Run historyWhether the workflow run succeeded, failed, timed out, or is waiting, plus step detailsUse when the workflow started but behaved incorrectly.

What to inspect

  • Run status such as Succeeded, Failed, Timed out, or Waiting.
  • Action-level inputs, outputs, and retry details.
  • Whether an action was Skipped because its runAfter criteria were not met.
  • Correlation IDs when working with support teams or external APIs.
Important: stateful workflows store run history by default. Stateless workflows do not store run history by default, although Standard can temporarily enable run history for stateless debugging when needed.

Stateful vs. Stateless Workflows

This decision matters most in Standard workflows. Stateful workflows persist run state externally for resiliency and longer-running processes. Stateless workflows keep state in memory for faster, shorter-lived executions.

DimensionStatefulStateless
Run historyStored by defaultNot stored by default
Best fitLong-running, auditable, resilient orchestrationsShort, high-throughput, in-memory processing
Trigger supportBroad trigger support including recurrence and managed polling triggersPush triggers only; no Recurrence or managed connector polling triggers
Operational tradeoffMore durable but more storage overheadFaster but less suited for large payloads and long durations

Key takeaway

Stateful workflows are the better fit for long-running monitored processes, while stateless workflows fit lightweight high-frequency tasks.

Example Use Cases

Choose stateful

Use stateful for document approvals, B2B message exchange, long-running order orchestration, delayed follow-up, large payload handling, and any workflow where run history, resumability, or auditability matter.

Choose stateless

Use stateless for short API mediation, lightweight webhook processing, fast request-response enrichment, or high-throughput message handling where the work usually completes within minutes and the payload stays relatively small.

Decision shortcut

If you need recurrence triggers, long waits, large content, asynchronous polling, or durable history, choose stateful. If you need the fastest in-memory execution with push triggers and short-lived processing, choose stateless.
