Logic Apps Structure

How workflows are packaged and deployed in Azure Logic Apps: what belongs to a workflow, what belongs to the Logic App resource, and how hosting model changes your deployment boundary.

Anatomy of a Logic App

A workflow is the executable sequence of one trigger and one or more actions. The Logic App resource is the deployment and management boundary where workflow definitions, connections, identity, and settings are configured.

In Standard, one Logic App resource can contain multiple workflows and relies on a storage account for state and artifacts. In multitenant Consumption, each resource hosts a single workflow.

What the Logic App Boundary Gives You

Everything inside one Logic App resource shares the same operational boundary, which is why structure decisions should match ownership, lifecycle, and runtime behavior.

Shared identity and networking

Managed identity, VNet integration model, and access controls are configured at the app resource level.

Shared app settings

Configuration values and runtime settings apply across workflows in the same Standard app.

Shared deployment lifecycle

Workflows in one Standard app are packaged and deployed together through one project boundary.

Shared runtime capacity

Workflows in the same Standard app use the same underlying worker and throughput envelope.

Connection governance

Connection references, authentication setup, and secret handling are managed with the app's environment model.

Monitoring context

Runs, diagnostics, and logs are easier to reason about when related workflows are grouped intentionally.

Design tip: Group workflows that share ownership, release cadence, and non-functional requirements. Split apps when teams, risk boundaries, or scaling profiles diverge.

Consumption vs Standard Structure Model

The deployment shape differs by hosting model. Consumption is one workflow per Logic App resource in multitenant infrastructure. Standard supports multiple workflows in one single-tenant Logic App resource.

  • Consumption: finer deployment granularity by default, but each workflow is its own app resource.
  • Standard: app-level packaging for multiple workflows with shared runtime, networking, and operations context.
  • Architecture implication: when teams need independent releases or isolation, separate app resources are usually the safer shape.

Structuring for Deployment

  • Align structure to ownership. Put workflows maintained by the same team in the same app when they share standards and release governance.
  • Separate by blast radius. Isolate business-critical paths from experimental or high-change workflows.
  • Separate by scaling profile. Keep high-volume and latency-sensitive workflows away from low-frequency batch orchestration.
  • Model environments consistently. Keep app settings and connection references parameterized so dev/test/prod use the same deployment artifact shape.
  • Use Monitoring as a design signal. If alerts and troubleshooting become noisy, split the app boundary to restore operational clarity.
