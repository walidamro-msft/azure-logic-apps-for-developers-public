Choose among Consumption, three Standard hosts, and Automation Project preview by evaluating tenancy, infrastructure ownership, networking, scaling, and cost. Pricing examples use public East US retail rates in USD.

Rule of thumb: Consumption and Logic Apps Automation support stateful workflows only. Use Workflow Service Plan, ASE v3, or Hybrid when the solution requires stateless workflows. For stateful workloads, use Consumption for simple bursty integration, Workflow Service Plan for normal managed Standard workloads, ASE v3 for dedicated App Service isolation, Hybrid for customer-managed Kubernetes, and Logic Apps Automation only after validating its preview terms.

How Consumption Executions Are Calculated

Consumption meters an operation each time it executes, even when the overall workflow fails. Duration and memory do not enter the calculation.

Monthly operation executions = trigger executions + action executions + loop iterations + retry executions

Polling triggers

Every poll is an execution, including checks that find no data and appear as Skipped. A one-minute poll can produce about 43,800 monthly trigger executions.

Push triggers

For webhook or event triggers, executions usually track received events. With Split On, each array item creates a separate workflow instance.

Loops

The loop action and actions inside it execute for each cycle. Ten items with one inner action produce 11 operations: 1 loop action + 10 inner executions.

Retries

The original attempt and every retry count. One action plus five retries produces six executions, whether the final result succeeds or fails.

Worked Example

A workflow runs 100,000 times per month. Each run uses three built-in actions and two Standard connector actions. Its push trigger is built in and fires once per run.

Built-in = 100,000 triggers + (100,000 runs × 3 actions) = 400,000 Standard connector = 100,000 runs × 2 actions = 200,000

The 4,000-operation grant applies only to built-in operations, so 396,000 built-in executions are billable. Managed connector executions do not consume that grant.

Architect Cheat Sheet

Choose Consumption when...

Traffic is intermittent, the workflow is simple, shared networking is acceptable, and per-execution cost stays below reserved compute.

Choose Standard when...

You need private networking, multiple workflows, stateful/stateless choice, local debugging, regional data residency, or predictable capacity.

Count polls, not only runs

A polling trigger is metered even when it finds nothing. Use trigger history and Azure Monitor metrics to measure actual execution volume.

Expand loops and retries

Actions inside loops and repeated attempts are separate executions. Model the expanded path, not just the number of designer cards.

Prefer built-in on Standard

Built-in connector operations run in-process and are included in compute; shared managed connector operations remain separately billed.

Price storage separately

Consumption retention uses a Logic Apps meter. Standard stateful workflows use your Azure Storage account and its transaction/capacity rates.
