Security & Limits

Secure the platform first, then shape workflows to live comfortably inside its runtime and connector limits.

SecurityGovernance

Security and Compliance Basics

  • Data is encrypted at rest and in transit.
  • Authentication choices include Entra ID, OAuth, API keys, and connector-specific mechanisms.
  • Microsoft positions Logic Apps for regulated industries through Azure's broader compliance portfolio.
Start with identity: use managed identity for Microsoft Entra-protected Azure resources when the selected trigger or action supports it. Store a credential only when identity-based authentication is unavailable.

Use Managed Identity End to End

A managed identity removes stored access tokens and credentials from workflow configuration, but enabling the identity is only the first step. The target resource must also grant that principal the minimum required data-plane role or access policy, and the connector must support managed identity authentication.

Hosting modelIdentity behaviorOperational implication
ConsumptionEnable either the system-assigned identity or one user-assigned identity for workflow use.Choose a user-assigned identity when the identity and its role assignments must survive logic app replacement.
StandardThe system-assigned identity is enabled by default. The app can also have multiple user-assigned identities, while each connection selects the identity it uses.Do not disable the default system identity without checking managed API connections; doing so changes its object ID and can break runtime access.
Standard HybridManaged identity authentication is not supported.Use an app registration and manage its credential lifecycle explicitly.
  1. Enable or attach the identity on the logic app resource.
  2. Assign the narrowest suitable role on the target resource, such as Storage Blob Data Contributor at the required container or account scope.
  3. Select managed identity authentication on the trigger, action, or connection and set the exact audience when the operation requires one.
  4. Test both design-time discovery and runtime access; they can require different permissions.

Manage Connections and Secrets as Dependencies

Prefer identity

Use managed identity and least-privilege RBAC before connection strings, keys, or user credentials.

Protect remaining secrets

Use Azure Key Vault references in Standard app settings where supported, and never commit local.settings.json.

Own the lifecycle

Document the owner, authentication type, target resource, rotation method, expiry signal, and recovery procedure for every connection.

Managed connectors create separate API connection resources. In Standard, each managed API connection also needs an access policy that permits the logic app identity to use the connection. Deploy the connection resource and access policy as infrastructure rather than relying on a designer-created production dependency.

Rotation warning: rotating a function key, certificate, client secret, or connector credential can break a workflow even when its definition has not changed. Monitor connection health and test rotation in a nonproduction environment.

Role-Based Access Control

RBAC should separate design authority, operational authority, and read-only access.

Developers

  • Edit workflow definitions and connections in controlled environments.
  • Need contributor-style rights, not broad subscription ownership.

Operators

  • Review runs, resubmit, inspect failures, and manage incidents.
  • Do not need permission to redesign workflows.

Auditors / Viewers

  • Read configuration, history, and diagnostics.
  • No mutation rights.

Limits Change Design

The deck dedicates a full section to limits because workflow shape is often a consequence of runtime ceilings: payload size, iteration counts, retention, duration, and connector behavior.

  • Large, long-running, or heavily branched workflows usually push you toward Standard and often toward stateful design.
  • High-concurrency designs must account for downstream throttling, not just Logic Apps runtime capability.
  • Stateless workflows trade history and payload flexibility for speed and reduced storage overhead.

Networking and Isolation

When the workflow touches private services or sensitive data, networking posture often becomes the deciding factor for Standard.

  • Use private endpoints and VNet integration where the environment requires private access paths.
  • Review firewall dependencies for every trigger and connector, especially managed connectors.
  • Document outbound dependencies early so network teams are not a late-stage blocker.

For multitenant Consumption firewall rules, prefer the LogicAppsManagement, LogicApps, and AzureConnectors service tags over copied IP lists where the network control supports service tags. For Standard, validate trigger and action FQDNs, App Service networking behavior, DNS resolution, and storage access before restricting traffic.

Protect Inbound Endpoints and Operational Data

  • Do not treat a Request trigger callback URL or SAS value as a complete authorization design. Use supported Microsoft Entra authorization policies, API Management, private endpoints, or another controlled ingress layer when callers need stronger authentication and policy enforcement.
  • Restrict inbound callers by identity, network, and protocol requirements appropriate to the hosting model.
  • Enable Secure Inputs and Secure Outputs on supported operations that process sensitive values, and inspect downstream actions for accidental re-exposure.
  • Keep secrets, tokens, personal data, and regulated payloads out of tracked properties, logs, alerts, and incident notifications.
  • Apply least-privilege RBAC and retention controls to run history, Log Analytics workspaces, storage archives, and Application Insights.

Make Governance Verifiable

Use Azure Policy, deployment policy checks, naming and tagging standards, resource locks where appropriate, and separate deployment identities to make the approved architecture repeatable. Review current limits during design and before load testing rather than copying numeric ceilings into long-lived standards.

  • Record data classification, region, retention, recovery objective, owner, and support path for each production workflow.
  • Alert on failed runs, failed triggers, authentication failures, abnormal volume, throttling, and expiring credentials.
  • Review role assignments, API connections, public network exposure, diagnostic destinations, and policy compliance on a schedule.
