Advanced Patterns

This section turns the workflow designer into a real orchestration tool: branch, repeat, retry, parallelize, and extend.

PatternsIntegration design

Control Flow: Conditions, Loops, and Scopes

Use conditions, switches, and loops to model the real business path rather than flattening every case into one linear sequence.

  • Condition: branch based on payload content, policy, or a prior action result.
  • For each / Until: repeat work over arrays or until a workflow reaches a target state.
  • Scope: group related work for readability, error handling, and modularity.

Trigger Strategy: Polling vs Push

Choosing the trigger style is an advanced design decision because it changes latency, connector load, and cost behavior.

Polling triggers

  • Use for systems that do not emit events reliably.
  • Tune recurrence carefully to balance freshness versus run frequency.

Push and webhook triggers

  • Use Request or Webhook when event producers can call back directly.
  • Prefer for lower latency and fewer unnecessary checks.
Important: If your workload can be event-driven, push-style triggers usually provide cleaner throughput and better cost efficiency.

Error Handling and Resilience

Reliable workflows assume transient failures will happen. Design explicit behavior for retries, compensations, and failure notifications.

Retry policies

Good for transient HTTP, connector, or downstream service failures.

Run-after rules

Route failed or timed-out actions into cleanup, alerting, or fallback logic.

Scopes as try/catch

Group a block of work and attach handling branches to its outcome.

Parallel Actions and Throughput

Independent work should run concurrently when order is not required. That shortens total run time, but concurrency must be balanced against destination limits and cost.

Integrating with External APIs

Logic Apps can call REST or SOAP services directly. Use this when no first-class connector exists or when a custom API surface gives better control.

  • Secure calls with Entra ID, OAuth, API keys, or gateway-managed policies.
  • Use API Management when you need centralized policy, observability, throttling, or external exposure.
  • Shape payloads before and after the call so your workflow does not leak partner-specific formats everywhere.

Content Types and Transformation Patterns

Advanced integrations often fail at boundaries between formats rather than transport. Normalize data early and keep transformations explicit.

  • Handle JSON, XML, binary, and multipart payloads deliberately at workflow boundaries.
  • Use workflow expressions to parse, reshape, and convert content between systems.
  • Define a canonical internal shape so downstream actions stay stable when partner schemas change.

Extend Workflows with Azure Functions and Service Bus

Logic Apps handles orchestration well. A Standard workspace can package local .NET functions with its workflows, with execution included in the Logic Apps cost model. A separately hosted Azure Function App adds independently deployed and scaled custom code, and follows its own Azure Functions pricing model. Service Bus adds durable asynchronous messaging for decoupled, multi-step processes.

Use functions when

  • Use a local function when reusable .NET code should remain in the Standard Logic Apps workspace and deployment.
  • Use an external Function App when the code needs independent deployment, reuse, scaling, or a separate lifecycle.

Use Service Bus when

  • You need queues or topics between systems.
  • You want reliable handoff between workflow stages or consumers.

On-Premises Integration

The on-premises data gateway remains the standard bridge for hybrid data access. Standard also broadens deployment options through single-tenant hosting and hybrid deployment paths, but gateway-based integration is still the common teaching path for enterprise workshops.

Note: Treat hybrid access as an enabler, not the default architecture. Keep workflows cloud-centric and isolate what truly must stay on premises.

B2B and Enterprise Integration Patterns

For partner ecosystems and regulated integrations, Logic Apps supports Enterprise Integration Pack capabilities through integration accounts.

  • Model trading partners, agreements, schemas, maps, and certificates as managed artifacts.
  • Support common B2B standards such as AS2, X12, and EDIFACT for interoperable exchange.
  • Separate transport concerns from transformation and validation for clearer support boundaries.

Reliability and Disaster Recovery Patterns

Mission-critical workflows need explicit resiliency design beyond basic retry settings.

Zone redundancy

Use zone-redundant hosting where available to reduce single-zone failure risk.

Cross-region strategy

Plan failover and recovery for integration accounts and dependent services.

Health-first operations

Use health checks and telemetry to detect degradation before end-user impact.
