Monitoring & Troubleshooting

Operate workflows with evidence from Azure Monitor, run history, diagnostic logs, alerts, and plan-appropriate telemetry.

Azure Monitor as the Operations Baseline

Start with Azure Monitor for platform metrics, alerting, diagnostic routing, and cross-resource analysis. Keep each telemetry surface distinct so you know what is collected automatically and what requires configuration.

SurfaceCollectionUse it for

Run and trigger historyStored for Consumption and Standard stateful workflowsInvestigating one execution, including trigger or action status, duration, inputs, and outputs.

  • Measure latency, throughput, and workflow run distribution.
  • Correlate workflow evidence with dependent APIs, functions, messaging services, and platform changes.

Configure Diagnostic Settings

Resource logs are not retained or queryable until you create a diagnostic setting. On the logic app resource, open Monitoring > Diagnostic settings, add a setting, and select the workflow runtime log category and AllMetrics.

Consumption

Select Workflow runtime diagnostic events.

Standard

Select Workflow Runtime Logs.

Destinations

Send data to Log Analytics, archive it to Storage, stream it through Event Hubs, or route it to a supported partner solution.

Important: Azure Monitor resource logs use a store-and-forward architecture. They can arrive several minutes late and are not guaranteed to be completely lossless, so do not use them as a transactional audit ledger.

Metrics, Logs, and Run History

Metrics

Use them to spot rates, saturation, spikes, and sustained failure patterns.

Resource logs

Use runtime events and tracked properties for queries, trends, correlation, and long-term routing.

Run history

Use it to inspect the status, duration, inputs, outputs, exceptions, retries, and connector responses for one execution.

Workflow typeRun history behaviorOperational note
ConsumptionStored by defaultReview run and trigger history from the logic app Overview page.
Standard statefulStored by defaultReview history at the workflow level, not only at the Logic App resource level.
Standard statelessNot stored by defaultEnable run history temporarily for debugging when necessary, then disable it because storage adds a performance cost.
Retention: Run history defaults to 90 days. Confirm the configured retention against audit requirements, and route diagnostic data to an appropriate long-term destination when evidence must be kept longer.

Application Insights for Standard Workflows

Application Insights enhanced telemetry is an optional capability for Standard logic apps when the subscription and app settings support it. It provides near-real-time Live Metrics, trace diagnostics, trigger and action events, retry visibility, exceptions, and workflow metrics. Consumption workflows should use Azure Monitor platform metrics, diagnostic settings, run history, and Log Analytics instead.

Note: Application Insights can help connect "what failed" with related application behavior, but end-to-end correlation depends on consistently propagating operation or client tracking identifiers across workflow and service boundaries.

Build an Actionable Alert Baseline

Create Azure Monitor metric or log alerts and connect them to action groups for email, SMS, push notifications, ITSM integration, webhooks, or automated remediation. Start with symptoms that require action rather than alerting on every event.

SignalWhy it mattersExample response
Failed runs or failure rateDetects broken workflows and downstream failures.Notify the owning team and link to run history or a Log Analytics query.
Failed triggersDetects authentication, endpoint, and polling failures before work starts.Check trigger outputs, connection health, and the downstream endpoint.
Latency or run durationDetects degradation before timeouts or backlogs become failures.Inspect slow actions, retries, throttling, and dependency latency.
Abnormal run volumeDetects silent inactivity, trigger duplication, and unexpected load.Compare against expected schedules and business volume.
Throttling or waiting runsDetects connector limits, concurrency pressure, and capacity constraints.Review retry behavior, concurrency, connector limits, and hosting capacity.
Plan difference: Alert signal names differ between Consumption and Standard. Select signals from the deployed resource's metric namespace rather than copying a rule name from another hosting model.

Make Runs Searchable

Add business-safe tracked properties to important actions so operators can search by identifiers such as an order, shipment, tenant, or message ID. Use a consistent client tracking ID or correlation ID across workflows and downstream services.

  • Choose identifiers that support an operator's investigation without exposing payload contents.
  • Propagate the same identifier through HTTP headers, messages, and dependent services where supported.
  • For B2B workflows, consider AS2, X12, or custom tracking schemas for business-message tracking.
Important: Tracked properties are telemetry. Never place secrets, credentials, access tokens, personal data, or regulated payloads in them.

Protect Operational Data

Run history can contain request bodies, response bodies, headers, secrets, and personal data. Treat monitoring access and retention as part of the workflow's security design.

  • Enable Secure Inputs and Secure Outputs on supported triggers and actions that handle sensitive values.
  • Check downstream actions too. Obscuring one operation does not guarantee that later operations will hide the same value.
  • Use least-privilege Azure RBAC and restrict access to run-history content and monitoring destinations.
  • Keep secrets in Azure Key Vault or secured parameters rather than hard-coding them in workflow definitions.
  • Set retention and archive access policies that match privacy, security, and audit requirements.

Structured Troubleshooting Flow

  1. Confirm the workflow and trigger are enabled, then distinguish Skipped, Failed, Waiting, Cancelled, and Aborted states.
  2. Review trigger history. A skipped polling trigger might simply mean that no data met its criteria, while a failed trigger includes error details in its outputs.
  3. Open the affected run and inspect the first failed or unexpectedly skipped action, including its inputs, outputs, retries, status code, duration, and tracking ID.
  4. Validate connection health, credentials, managed identity permissions, connector limits, and downstream availability.
  5. Review metrics and alerts for sustained failure rates, throttling, latency changes, abnormal volume, or waiting runs.
  6. Check the Azure Activity Log for configuration changes and Azure Resource Health or Service Health for platform incidents.
  7. If expected logs are absent, confirm the diagnostic setting, selected categories, destination, and ingestion delay.
  8. Classify the issue as workflow logic, data, dependency, capacity, permissions, configuration, transient failure, or Azure platform health before choosing remediation.

Recover Without Creating a Second Incident

  • Cancel a stuck or obsolete run when it blocks concurrency or no longer has business value.
  • Resubmit a completed Consumption or Standard stateful run only after checking whether downstream effects are safe to repeat.
  • Remember that resubmission uses the workflow version from the original run, even if the current workflow definition has changed.
  • Use rerun-from-action only when the workflow structure and action support it; parallel paths, loops, and other complex control flow have limitations.
  • Record the original run ID, recovery decision, and resulting run ID when an audit trail is required.
Important: Resubmission can create duplicate records, repeat notifications, or retry a destructive operation. Confirm idempotency or use a deduplication key before rerunning production work.

Build Reliable Workflows

  • Use retry policies only where the operation is safe to retry.
  • Use timeouts, scopes, and run after paths to separate transient recovery from terminal failure handling.
  • Add conditional checks before expensive or destructive downstream actions.
  • Route failures into explicit logging, notification, or compensating actions.
  • Design idempotent downstream operations because Azure Logic Apps uses at-least-once message delivery and rare duplicate delivery is possible.
  • Monitor queues and dead-letter destinations when a messaging service participates in the workflow.
