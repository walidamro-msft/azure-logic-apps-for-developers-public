Connectors and Integration Accounts

Choose where an integration runs, how it is billed and secured, and whether the solution needs reusable B2B artifacts.

Design guidanceConnector strategyB2B integration

Built-In Connectors

Built-in connectors run natively within your Logic Apps compute environment. General built-ins such as HTTP, Request, Schedule, and Data Operations implement workflow behavior directly in the runtime. Many Standard service-provider built-ins run in the same process as the single-tenant runtime, avoiding the extra network hop to shared connector infrastructure and generally providing lower latency, higher throughput, private-network access, and compute-inclusive execution compared with managed equivalents.

Plan check: the built-in gallery differs by workflow type. Consumption has a smaller native gallery. Most service-provider connectors in this list require Standard, and stateless Standard workflows support push triggers only.

Built-In Cost Model

Hosting modelHow built-in operations are billedCost drivers to model
ConsumptionAn initial monthly grant of built-in executions is shared per Azure subscription. Executions above that grant follow Consumption Actions pricing.Trigger checks, action executions, loop cycles, retries, and each workflow instance created by Split On.
StandardBuilt-in operations are included without a separate per-operation charge, but the Standard hosting capacity is billed while provisioned.Selected hosting tier and instances, scaling, workflow storage transactions, networking, and any downstream Azure service charges.
Cost boundary: included connector execution does not make the connected service free. External Azure Function Apps, Storage, Service Bus, Event Hubs, SQL, Key Vault, Cosmos DB, data transfer, and other dependencies retain their own pricing. Local functions packaged inside a Standard Logic Apps workspace are included in the Logic Apps hosting cost.

Managed Connectors

Managed connectors are Microsoft-managed services hosted in global, multitenant Azure. They appear under Shared in the designer and usually require an API connection plus authentication. Because a managed call travels from the workflow runtime to shared connector infrastructure before reaching the target service, managed connectors generally have higher latency than an equivalent built-in connector. Both Consumption and Standard workflows can use them, but connector availability, regional support, authentication choices, throttling, and pricing must be checked for the selected operation.

Connector Namespace boundary: Azure Connector Namespace is currently in preview and provides access to the managed connector catalog only. It does not expose Logic Apps built-in connectors, which run natively within a Logic Apps compute environment.
Architecture boundary: a Standard workflow can run on single-tenant compute while a managed connector call still leaves that runtime and executes through the shared managed connector service. Include the connector's network path, connection resource, access policy, throttling, latency, and per-call meter in the design.

Managed Connector Cost Model

Hosting modelHow managed operations are billedCost drivers to model
ConsumptionEach execution follows the connector's Standard or Enterprise Consumption pricing class.Polling checks, action executions, retries, loops, Split On workflow instances, and connector class.
StandardEach call follows the same Standard or Enterprise connector pricing classes in addition to Standard hosting cost.Every connector call, including calls generated by pagination or chunking, retries, loops, and polling.
Gateway note: the on-premises data gateway resource does not itself incur a Logic Apps charge. Operations routed through it are still billed using the workflow and managed connector pricing model, and you remain responsible for the machine that hosts the gateway.

Integration Accounts

An Integration Account is a separate Azure resource outside the Logic App resource and its workflows. It provides a reusable container for business-to-business artifacts, including trading partners, agreements, schemas, maps, certificates, and custom metadata. The Logic App references or links to the Integration Account; the account is not contained inside the Logic App.

Resource boundary: the Logic App and Integration Account are peer Azure resources. They can be deployed and managed independently, and multiple Logic Apps can reuse a centrally governed Integration Account when the applicable subscription, region, and workflow requirements are met.

Consumption workflows must link the Logic App resource to the Integration Account before using B2B artifacts. Standard can store maps and schemas directly in the Logic App project, while partners and agreements still require an Integration Account; some Standard operations connect to the account rather than requiring a resource link.

Cost componentHow it is billedDesign implication
Integration Account resourceBilled separately according to the selected Integration Account tier in both Consumption and Standard.Choose the tier from required artifact capacity, agreements, partners, and B2B capabilities rather than workflow hosting type.
B2B workflow operationsConsumption operations follow their applicable execution meter. Supporting servicesLogic Apps hosting, workflow storage, Key Vault, networking, monitoring, and data transfer are billed separately where applicable.An Integration Account tier is only one component of the total B2B solution cost.
Cost optimization: Standard workflows can store maps and schemas directly in the logic app resource. If the solution does not require centrally governed partners, agreements, certificates, or cross-resource artifact reuse, project-level artifacts can avoid an unnecessary Integration Account resource.
